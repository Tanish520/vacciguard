Component Guide

This section explains each major component in simple terms and shows how one component supports another. It is meant for readers who want to understand the logic of the architecture without already knowing the tools.

What It Does What It Receives Who It Helps
Source Layer

Live Telemetry Simulator

This component acts like many vaccine refrigerators sending readings continuously. It is the starting point of the real-time pipeline.

  • Creates repeatable event streams for normal, burst, and failure experiments.
  • Helps Kafka by giving it realistic event traffic to ingest.
  • Helps the evaluation phase because the same workloads can be replayed consistently.
Ingestion Layer

Kafka

Kafka is the event buffer between the source and the processor. It does not do the business logic itself. Its job is to hold and organize incoming live events safely.

  • Receives telemetry from the simulator.
  • Helps Spark by feeding it events in a durable and organized way.
  • Helps scaling because Kafka lag shows when the system is falling behind.
Processing Layer

Spark Structured Streaming

This is the brain of the live pipeline. It turns raw events into meaningful outputs.

  • Receives events from Kafka.
  • Helps Redis by producing the latest cleaned operational state.
  • Helps S3 by writing processed history for later analytics.
Support Data

Lookup Data

Lookup data gives meaning to raw device events. Without it, the system only sees IDs and numbers, not facilities, regions, or allowed ranges.

  • Helps Spark classify whether a reading is safe or unsafe.
  • Helps reports by enabling district, state, and facility grouping.
  • Helps the live path and the batch path in the same way.
Hot Storage

Redis

Redis stores only the most recent and operationally useful information. It is designed for fast access, not for long-term history.

  • Receives processed current-state updates from Spark streaming.
  • Helps dashboards or live views respond quickly.
  • Helps latency by avoiding heavy historical queries for current status.
Cold And Analytical Storage

S3 + Parquet

This is the long-term memory of the system. It keeps raw files and curated historical outputs for analytics and reports.

  • Receives raw batch files and processed telemetry history.
  • Helps Spark batch by providing historical data for deeper analysis.
  • Helps cost efficiency by storing large history more cheaply than hot storage.
Workflow Orchestration

Airflow

Airflow decides when batch work should happen. It does not replace Spark. It schedules Spark jobs, backfills, and repeatable experiment runs.

  • Helps batch processing start at the right time.
  • Helps evaluation by making runs repeatable.
  • Helps cost control by not running heavy jobs continuously.
Elastic Control

KEDA, HPA, And Spark Dynamic Allocation

These controls make the architecture elastic. They let the system grow during pressure and shrink when demand falls.

  • KEDA helps Kafka-related workers scale using backlog signals.
  • HPA helps normal services scale using CPU or memory signals.
  • Spark dynamic allocation helps the processing layer add or remove executors.
Observability Layer

Prometheus + Grafana

These tools make the system understandable during experiments. They show what is happening instead of forcing you to rely only on logs.

  • Prometheus collects metrics from Kafka, Spark, Airflow, and Kubernetes.
  • Grafana helps humans see throughput, lag, delay, failures, and recovery visually.
  • They help the report because claims can be backed with graphs.

How Components Help Each Other In The Live Path

The real-time path works because each component reduces the burden on the next one.

Simulator -> Kafka The simulator creates live traffic, and Kafka safely absorbs that traffic even if processing is temporarily slower.
Kafka -> Spark Streaming Kafka gives Spark a durable and ordered event source so Spark can focus on logic instead of direct device connectivity.
Lookup Data -> Spark Streaming Lookup data gives Spark the context needed to turn raw numbers into facility-aware breach decisions.
Spark Streaming -> Redis Spark writes only the latest important state to Redis, so live monitoring stays fast.
Spark Streaming -> S3 Spark also writes processed records to the historical layer so the live path and reporting path stay connected.

How Components Help Each Other In The Batch And Evaluation Path

The batch and control layers turn the pipeline into a complete cloud project instead of only a streaming demo.

Batch Logs -> S3 Raw -> Airflow Batch files land safely first, and Airflow later decides when they should be processed.
Airflow -> Spark Batch Airflow triggers repeatable Spark jobs so historical analytics and benchmarks can run consistently.
S3 Curated -> Spark Batch The historical telemetry already stored in S3 helps the batch layer produce compliance summaries and trend analysis.
Kafka + Spark -> KEDA / Dynamic Allocation Lag and workload signals from the running pipeline help the control layer decide when to scale up or down.
All Runtime Components -> Prometheus -> Grafana Each component emits signals that Prometheus collects, and Grafana turns those signals into graphs for analysis.

How The System Works

This view focuses on behavior over time. Instead of showing all components at once, it breaks the system into four easy stories: live telemetry, batch analytics, elastic scaling, and failure recovery.

Real-Time Path Batch Path Elastic Scaling Recovery Logic

Real-Time Telemetry Flow

What happens when live fridge events arrive.

1
Simulator sends eventsTemperature, battery, door state, and timestamps are published continuously.
2
Kafka receives the streamEvents enter the ingestion layer and can wait safely during short bursts.
3
Spark validates and deduplicatesBad records are filtered and duplicate event IDs are suppressed.
4
Lookup data enriches recordsThe pipeline adds facility, district, state, and threshold context.
5
Breach logic runsThe system decides whether the temperature is safe or unsafe.
6
Redis updates live stateCurrent device status becomes quickly available for operational views.
7
S3 stores historyProcessed records are also written to the historical analytics layer.

Batch Analytics Flow

What happens when scheduled files arrive for deeper analysis.

1
Daily logs are uploadedOperations or maintenance files arrive as batch input.
2
S3 raw stores the uploadThe raw version is preserved for auditability and replay.
3
Airflow triggers a runThe batch workflow starts on schedule or during a controlled benchmark.
4
Spark batch combines datasetsBatch logs, lookup data, and telemetry history are processed together.
5
Reports are producedThe system creates compliance summaries and historical analysis outputs.
6
Curated outputs are storedFinal analytical results are written back to S3 in Parquet format.

Elastic Scaling Flow

What happens when traffic grows suddenly and the system must adapt.

1
Traffic spike beginsThe simulator pushes more events than usual.
2
Kafka backlog growsWaiting messages increase, which shows that the system is under pressure.
3
KEDA and HPA reactRelevant ingestion-side workers scale out when lag or resource pressure rises.
4
Spark adds executorsDynamic allocation increases processing-side compute capacity.
5
Lag starts droppingHigher consumption and higher processing capacity reduce the backlog.
6
System scales back downWhen the load returns to normal, extra pods and executors are released.

Failure and Recovery Flow

What happens when a component crashes or becomes unavailable.

1
A component failsA pod crashes, a service stops, or a processing task is interrupted.
2
Kubernetes restarts itThe platform attempts to restore the failed component automatically.
3
Kafka keeps the eventsUnprocessed records remain available for replay instead of disappearing.
4
Spark resumes from checkpointsThe stream job recovers progress instead of starting from scratch.
5
Processing continuesThe pipeline returns to normal operation and metrics capture recovery time.

Full Working Story

Live events enter continuously Telemetry from simulated cold-chain units is published into Kafka throughout the day.
The stream path handles urgent logic Spark validates, deduplicates, enriches, and checks every live event for breach conditions.
The hot store serves current status Redis keeps the latest state ready for fast operational views.
The historical layer keeps long-term value S3 and Parquet preserve processed records for reporting and compliance analytics.
The batch path adds depth Daily operations or maintenance logs are processed on schedule for summaries and historical reports.
The control layer makes it measurable Scaling, observability, and reproducibility are built into the architecture instead of being added later.
